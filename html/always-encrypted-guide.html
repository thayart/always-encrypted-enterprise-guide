Always Encrypted
คู่มือฉบับสมบูรณ์สำหรับ SQL Server

คู่มืออ่านแล้วลงมือทำตามได้จริง — ตั้งแต่แนวคิด CMK/CEK สถาปัตยกรรมการเข้ารหัสฝั่งไคลเอนต์ ไปจนถึงคำสั่ง T-SQL/PowerShell ตัวอย่างโค้ด และแนวปฏิบัติด้านการจัดการคีย์ระดับองค์กร

SQL Server 2016+ Azure SQL Database Secure Enclaves (2019+) Azure Key Vault / HSM
ส่วนที่ 1 — หลักการพื้นฐาน

01ภาพรวมและแนวคิดพื้นฐาน

Always Encrypted เป็นฟีเจอร์ของ SQL Server (2016+) และ Azure SQL Database ที่เข้ารหัสข้อมูลอ่อนไหว ที่ฝั่งไคลเอนต์ ก่อนส่งไปยังฐานข้อมูล คีย์เข้ารหัสไม่เคยถูกส่งหรือเก็บบนเซิร์ฟเวอร์ ทำให้แม้แต่ DBA หรือผู้ที่เข้าถึงเซิร์ฟเวอร์โดยตรงก็เห็นได้เพียง ciphertext เท่านั้น

แนวคิดหลัก การเข้ารหัส/ถอดรหัสเกิดขึ้นที่ driver ฝั่งไคลเอนต์ เท่านั้น (ADO.NET / JDBC / ODBC) — SQL Server engine ไม่มีทางเห็นข้อมูลต้นฉบับได้เลย นี่คือหัวใจของ separation of duties

ปัญหาที่แก้ไข เทียบกับ TDE

Transparent Data Encryption (TDE) เข้ารหัสเฉพาะไฟล์บนดิสก์ (at rest) แต่เมื่อข้อมูลถูกอ่านเข้าหน่วยความจำของเซิร์ฟเวอร์ จะกลายเป็น plaintext ทันที — ผู้มีสิทธิ์ sysadmin ยังเห็นข้อมูลจริงได้ Always Encrypted ปิดช่องว่างนี้โดยทำให้ข้อมูล อยู่ในรูปเข้ารหัส ตลอดวงจรชีวิต: ดิสก์ หน่วยความจำฝั่งเซิร์ฟเวอร์ เครือข่าย และ backup — ควรใช้ ร่วมกับ TDE และ TLS ไม่ใช่แทนกัน

เทคนิคเข้ารหัสที่ไหนป้องกัน DBA เห็นข้อมูล
TDEไฟล์ฐานข้อมูล/log/backupไม่
Cell-level Encryptionคอลัมน์ (เขียนโค้ดเรียกฟังก์ชันเอง)บางส่วน
Always Encryptedคอลัมน์ ตั้งแต่ฝั่งไคลเอนต์ใช่
+ Secure Enclavesคอลัมน์ + คำนวณใน enclave ฝั่งเซิร์ฟเวอร์ใช่ (ยกเว้นใน enclave)

ศัพท์ที่ต้องรู้ก่อนเริ่ม

CMK
Column Master Key — กุญแจแม่ เก็บนอก SQL Server
CEK
Column Encryption Key — เข้ารหัสข้อมูลจริงในคอลัมน์
Deterministic
ciphertext เดิมทุกครั้ง → ค้นหา/join ได้
Randomized
ciphertext สุ่มทุกครั้ง → ปลอดภัยกว่า ค้นหาไม่ได้

กรณีใช้งานทั่วไป

  • ระบบที่ต้องปฏิบัติตาม PCI-DSS (เลขบัตรเครดิต), HIPAA (ข้อมูลสุขภาพ), PDPA/GDPR (ข้อมูลส่วนบุคคล)
  • SaaS แบบ multi-tenant ที่ผู้ให้บริการ cloud ต้อง "เห็นข้อมูลน้อยที่สุด"
  • องค์กรที่ใช้ DBA ภายนอก (outsourced) แต่ไม่ต้องการให้เห็นข้อมูลลูกค้า
  • ย้ายฐานข้อมูลขึ้น Azure SQL แต่ต้องการควบคุมคีย์เข้ารหัสเอง (customer-managed keys)

02สถาปัตยกรรมและองค์ประกอบหลัก

ลำดับการทำงานเมื่อแอปพลิเคชันรัน query:

  1. ส่ง parameterized query — แอปส่ง query ผ่าน driver พร้อมพารามิเตอร์ (ไม่ใช่ dynamic SQL)
  2. Driver ตรวจ metadata — เรียก sp_describe_parameter_encryption เพื่อรู้ว่าพารามิเตอร์ใดตรงกับคอลัมน์เข้ารหัส
  3. เข้ารหัสที่ client — driver ดึง CMK จาก key store, ถอดรหัส CEK, ใช้ CEK เข้ารหัสค่าพารามิเตอร์เป็น ciphertext
  4. SQL Server ประมวลผล — รับ ciphertext และเปรียบเทียบแบบ equality โดยไม่รู้ค่า plaintext เลย
  5. ถอดรหัสผลลัพธ์ — driver ถอดรหัสแถวผลลัพธ์ด้วย CEK ก่อนคืนค่าให้แอปพลิเคชันเป็น plaintext

Column Master Key (CMK)

CMK คือกุญแจแม่ที่ปกป้อง CEK — ไม่ถูกเก็บใน SQL Server เลย เซิร์ฟเวอร์รู้เพียง metadata (ชื่อ/ตำแหน่ง/ประเภท key store) การขโมยไฟล์ฐานข้อมูล (.mdf/.bak) เพียงอย่างเดียวจึง ไม่เพียงพอ ต่อการถอดรหัส

Key Storeเหมาะกับหมายเหตุ
Windows Certificate StoreDev/Test, องค์กรเล็กต้อง export/import .pfx ทุกเครื่อง
Azure Key VaultProduction, Azure SQL, HybridRBAC, audit log, rotation, HSM-backed
Hardware Security ModuleFIPS 140-2/3 complianceต้องมี HSM on-premises + CNG provider
Windows CSP/CNGCustom PKI, smart cardผูกกับ enterprise CA

Column Encryption Key (CEK)

CEK เข้ารหัสข้อมูลจริงในคอลัมน์ SQL Server เก็บเฉพาะ CEK ที่ถูกเข้ารหัสด้วย CMK แล้ว ใน sys.column_encryption_keys หนึ่ง CEK สามารถถูกเข้ารหัสด้วยได้หลาย CMK (รองรับ key rotation) และหลายคอลัมน์สามารถใช้ CEK ร่วมกัน หรือแยกตามความอ่อนไหวของข้อมูล

ถ้าต้องการทั้งสองอย่าง การค้นหาแบบ range/pattern และ ความปลอดภัยสูงพร้อมกัน ให้พิจารณา Secure Enclaves (บทที่ 9) ซึ่งรองรับ LIKE/range query บน randomized encryption ได้

Metadata ที่ใช้ตรวจสอบบ่อย

T-SQL
SELECT
    t.name AS table_name,
    c.name AS column_name,
    c.encryption_type_desc,
    c.encryption_algorithm_name,
    cek.name AS cek_name
FROM sys.columns c
JOIN sys.tables t ON c.object_id = t.object_id
JOIN sys.column_encryption_keys cek
    ON c.column_encryption_key_id = cek.column_encryption_key_id
WHERE c.encryption_type IS NOT NULL;

อัลกอริทึมที่ใช้คือ AEAD_AES_256_CBC_HMAC_SHA256 — Authenticated Encryption ที่ให้ทั้งความลับ (confidentiality) และตรวจสอบความถูกต้อง (integrity) ป้องกันการปลอมแปลง ciphertext โดยไม่ถูกตรวจจับ

Checklist ก่อนเริ่มติดตั้ง

  • ทำ Data Classification ระบุคอลัมน์อ่อนไหว (PII, การเงิน, สุขภาพ, credential)
  • ตัดสินใจ Deterministic/Randomized ต่อคอลัมน์ — ดูว่าคอลัมน์ถูกใช้ใน WHERE/JOIN/index หรือไม่
  • ตัดสินใจว่าต้องใช้ Secure Enclave หรือไม่ (range query, in-place rotation)
  • ออกแบบกลยุทธ์ key store (Certificate Store / Azure Key Vault / HSM)
  • ประเมินผลกระทบต่อแอปพลิเคชัน — driver, dynamic SQL, ORM, reporting tools
  • ยืนยันสิทธิ์ที่จำเป็น: ALTER ANY COLUMN MASTER KEY, ALTER ANY COLUMN ENCRYPTION KEY
ข้อจำกัดชนิดข้อมูล ไม่รองรับ xml, text, ntext, image, rowversion, sql_variant, geography, geometry คอลัมน์ deterministic ต้องมี collation ลงท้าย _BIN2 เช่น Latin1_General_BIN2

สภาพแวดล้อมทดสอบที่แนะนำ

Dev → Staging (ขนาดข้อมูลใกล้เคียง production เพื่อวัดเวลา encryption/rotation จริง) → มีแผน rollback + backup เสมอ ก่อนดำเนินการใน production เพราะการเข้ารหัสคอลัมน์เป็น operation ที่ใช้เวลาและ lock ตาราง

ส่วนที่ 2 — การติดตั้งใช้งาน

04การติดตั้งผ่าน SSMS (Wizard)

เหมาะสำหรับการติดตั้งครั้งแรก ทดสอบ หรือฐานข้อมูลขนาดเล็ก-กลาง คลิกขวาที่ฐานข้อมูลใน Object Explorer → Tasks → Encrypt Columns...

  1. Column Selection — เลือกตาราง/คอลัมน์ที่จะเข้ารหัส และเลือก Encryption Type (Deterministic/Randomized) ต่อคอลัมน์ ถ้าเป็น primary key/unique/index ต้องเป็น Deterministic เท่านั้น
  2. Master Key Configuration — สร้าง CMK ใหม่หรือเลือกที่มีอยู่ เลือก Key Store: Certificate Store, Azure Key Vault (sign-in ด้วยบัญชีที่มีสิทธิ์ Key Vault Crypto Officer), หรือ HSM
  3. Run Settings — เลือก "Proceed to finish now" (dev/test) หรือ "Generate PowerShell script to run later" (แนะนำสำหรับ production)
  4. Summary & Run — ตรวจสอบสรุปการตั้งค่าทั้งหมดแล้วกด Finish
สำหรับ Production เลือก "Generate PowerShell script" เสมอ เพื่อ review ก่อนรัน, รันนอกเวลาทำการ, และควบคุม transaction/locking ได้ดีกว่า

ตรวจสอบผลลัพธ์หลังเข้ารหัส

T-SQL
SELECT t.name AS table_name, c.name AS column_name,
       c.encryption_type_desc, c.encryption_algorithm_name
FROM sys.columns c
JOIN sys.tables t ON c.object_id = t.object_id
WHERE c.encryption_type IS NOT NULL
ORDER BY t.name, c.name;

จากนั้นต้องอัปเดต connection string ของทุกแอปพลิเคชันให้เปิด Column Encryption Setting=Enabled (ดูบทที่ 6) มิฉะนั้นแอปจะเห็นเพียง ciphertext แบบ binary

ข้อควรระวังกับตารางขนาดใหญ่ การเข้ารหัสข้อมูลที่มีอยู่แล้วคือ operation ที่อ่าน-เขียนทุกแถวใหม่ ใช้เวลานาน มี table lock — ควรทำนอกเวลาทำการและ backup ก่อนเสมอ สำหรับตารางใหญ่มากใช้ Online Encryption ผ่าน PowerShell แทน (บทที่ 5.2.3)

05การติดตั้งผ่าน T-SQL และ PowerShell

เหมาะสำหรับ Production, automation และ CI/CD เพราะ script, review, version control ได้

ข้อจำกัดสำคัญ T-SQL ล้วนสร้างได้แค่ metadata เท่านั้น — ไม่สามารถเข้ารหัสข้อมูลที่มีอยู่แล้วในตารางได้ เพราะ SQL Server ไม่มีสิทธิ์เข้าถึง CMK การเข้ารหัสข้อมูลจริงต้องผ่าน PowerShell module SqlServer

5.1 T-SQL — สำหรับตารางใหม่ที่ยังไม่มีข้อมูล

T-SQL
-- สร้าง Column Master Key (Azure Key Vault)
CREATE COLUMN MASTER KEY [CMK_AKV]
WITH (
    KEY_STORE_PROVIDER_NAME = N'AZURE_KEY_VAULT',
    KEY_PATH = N'https://mycompany-kv.vault.azure.net/keys/CMK-Prod/8f8f...'
);

-- สร้าง Column Encryption Key (ENCRYPTED_VALUE มาจาก PowerShell)
CREATE COLUMN ENCRYPTION KEY [CEK_Auto1]
WITH VALUES (
    COLUMN_MASTER_KEY = [CMK_AKV],
    ALGORITHM = 'RSA_OAEP',
    ENCRYPTED_VALUE = 0x016E000001630075...
);

-- สร้างตารางพร้อมคอลัมน์เข้ารหัส
CREATE TABLE dbo.Customer (
    CustomerId   INT IDENTITY PRIMARY KEY,
    NationalId   CHAR(13) COLLATE Latin1_General_BIN2
        ENCRYPTED WITH (
            COLUMN_ENCRYPTION_KEY = [CEK_Auto1],
            ENCRYPTION_TYPE = DETERMINISTIC,
            ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA256'
        ) NOT NULL,
    CreditCardNumber VARCHAR(19) COLLATE Latin1_General_BIN2
        ENCRYPTED WITH (
            COLUMN_ENCRYPTION_KEY = [CEK_Auto1],
            ENCRYPTION_TYPE = RANDOMIZED,
            ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA256'
        ) NULL,
    FullName NVARCHAR(100) NOT NULL  -- ไม่เข้ารหัส
);

5.2 PowerShell — สำหรับตารางที่มีข้อมูลอยู่แล้ว (แนะนำสำหรับ Production)

PowerShell
Install-Module -Name SqlServer -Scope CurrentUser -AllowClobber
Import-Module SqlServer

$connStr = "Server=myserver;Database=MyProdDb;Integrated Security=True;"
$database = Get-SqlDatabase -ConnectionString $connStr

# สร้าง CMK ผ่าน Azure Key Vault
$akvKey = New-SqlAzureKeyVaultColumnMasterKeySettings `
    -KeyURL "https://mycompany-kv.vault.azure.net/keys/CMK-Prod/8f8f..."
New-SqlColumnMasterKey -Name "CMK_Prod" -InputObject $database `
    -ColumnMasterKeySettings $akvKey

# สร้าง CEK
New-SqlColumnEncryptionKey -Name "CEK_Prod" -InputObject $database `
    -ColumnMasterKey "CMK_Prod"

# กำหนดการเข้ารหัสต่อคอลัมน์
$cesc1 = New-SqlColumnEncryptionSettings `
    -ColumnName "dbo.Customer.NationalId" `
    -EncryptionType "Deterministic" -EncryptionKey "CEK_Prod"
$cesc2 = New-SqlColumnEncryptionSettings `
    -ColumnName "dbo.Customer.CreditCardNumber" `
    -EncryptionType "Randomized" -EncryptionKey "CEK_Prod"

# รัน Online Encryption (ลด downtime บนตารางที่มีข้อมูลอยู่แล้ว)
Set-SqlColumnEncryption -ColumnEncryptionSettings $cesc1, $cesc2 `
    -InputObject $database -OnlineEncryption
CI/CD แยก pipeline step "provision key metadata" ออกจาก "encrypt data" เสมอ อย่าเก็บ ENCRYPTED_VALUE หรือ credential ของ Key Vault ไว้ใน source control — ใช้ pipeline secret variable หรือ managed identity แทน

06การพัฒนาแอปพลิเคชันให้รองรับ

หลักการสำคัญ: ต้องใช้ parameterized query เท่านั้น — dynamic SQL ที่ต่อ string ตรง ๆ ใช้งานไม่ได้กับ คอลัมน์เข้ารหัส และต้องระบุชนิด/ขนาดพารามิเตอร์ให้ตรงกับคอลัมน์เป๊ะ

.NET (Microsoft.Data.SqlClient)

C#
var connString =
    "Server=myserver;Database=MyProdDb;" +
    "Column Encryption Setting=Enabled;Integrated Security=True;";

using var connection = new SqlConnection(connString);
connection.Open();

using var command = new SqlCommand(
    "SELECT CustomerId, FullName FROM dbo.Customer WHERE NationalId = @NationalId",
    connection);

// ชนิด/ขนาดต้องตรงกับคอลัมน์เป๊ะ
command.Parameters.Add("@NationalId", System.Data.SqlDbType.Char, 13).Value = "1234567890123";

using var reader = command.ExecuteReader();
while (reader.Read())
    Console.WriteLine($"{reader["CustomerId"]}: {reader["FullName"]}");

JDBC (Java)

Java
String url = "jdbc:sqlserver://myserver;database=MyProdDb;columnEncryptionSetting=Enabled;";
try (Connection conn = DriverManager.getConnection(url, "user", "password");
     PreparedStatement pstmt = conn.prepareStatement(
         "SELECT CustomerId, FullName FROM dbo.Customer WHERE NationalId = ?")) {
    pstmt.setString(1, "1234567890123");
    try (ResultSet rs = pstmt.executeQuery()) {
        while (rs.next())
            System.out.println(rs.getInt("CustomerId") + ": " + rs.getString("FullName"));
    }
}

Python (pyodbc)

Python
import pyodbc

conn_str = (
    "Driver={ODBC Driver 17 for SQL Server};"
    "Server=myserver;Database=MyProdDb;"
    "ColumnEncryption=Enabled;Trusted_Connection=yes;"
)
conn = pyodbc.connect(conn_str)
cursor = conn.cursor()
cursor.execute(
    "SELECT CustomerId, FullName FROM dbo.Customer WHERE NationalId = ?",
    ("1234567890123",)
)
for row in cursor.fetchall():
    print(row.CustomerId, row.FullName)

Errorสาเหตุวิธีแก้
Operand type clashชนิด/ขนาดพารามิเตอร์ไม่ตรงกับคอลัมน์ตรวจ data type mapping
Encryption scheme mismatchใช้ range/LIKE บนคอลัมน์ที่ไม่มี enclaveปรับ query หรือใช้ Secure Enclave
Failed to decrypt CEKไม่มีสิทธิ์เข้าถึง CMK จาก client นั้นตรวจ certificate/Key Vault access policy
ได้ binary garbage แทนค่าจริงลืมเปิด Column Encryption Settingแก้ connection string

ORM ทั่วไปใช้งานได้ตามปกติ (Entity Framework Core, Dapper, Hibernate) ตราบใดที่ query ที่ generate ออกมาเป็น parameterized/PreparedStatement — ระวัง ORM บางตัวที่ inline literal แทน parameter ในบางกรณี (เช่น batch insert บางรูปแบบ)

JOIN บนคอลัมน์เข้ารหัส

SQL Server เปรียบเทียบ ciphertext ของสองคอลัมน์กันได้ตรง ๆ ที่ฝั่งเซิร์ฟเวอร์ (ไม่ต้องส่ง plaintext ผ่าน driver เลย) ก็ต่อเมื่อ ทั้งสองคอลัมน์เข้ากันได้ครบทุกข้อ:

  • ทั้งคู่เป็น Deterministic (Randomized join กันไม่ได้แม้แต่กับตัวเอง)
  • ใช้ CEK เดียวกัน
  • ใช้ algorithm เดียวกัน (AEAD_AES_256_CBC_HMAC_SHA256)
  • ชนิดข้อมูล/collation ตรงกัน (_BIN2)
T-SQL — Schema
CREATE TABLE dbo.Customer (
    CustomerId INT IDENTITY PRIMARY KEY,               -- ไม่เข้ารหัส
    NationalId CHAR(13) COLLATE Latin1_General_BIN2
        ENCRYPTED WITH (
            COLUMN_ENCRYPTION_KEY = [CEK_Prod],
            ENCRYPTION_TYPE = DETERMINISTIC,
            ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA256'
        ) NOT NULL,
    FullName NVARCHAR(100) NOT NULL
);

CREATE TABLE dbo.Orders (
    OrderId INT IDENTITY PRIMARY KEY,
    CustomerNationalId CHAR(13) COLLATE Latin1_General_BIN2
        ENCRYPTED WITH (
            COLUMN_ENCRYPTION_KEY = [CEK_Prod],         -- ต้องเป็น CEK เดียวกับ Customer.NationalId
            ENCRYPTION_TYPE = DETERMINISTIC,
            ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA256'
        ) NOT NULL,
    OrderDate DATETIME2 NOT NULL
);
C#
using var command = new SqlCommand(@"
    SELECT o.OrderId, o.OrderDate, c.FullName
    FROM dbo.Orders o
    JOIN dbo.Customer c ON o.CustomerNationalId = c.NationalId
    WHERE o.OrderDate >= @StartDate", connection);

command.Parameters.Add("@StartDate", System.Data.SqlDbType.DateTime2).Value = DateTime.Today.AddDays(-30);

using var reader = command.ExecuteReader();
while (reader.Read())
    Console.WriteLine($"{reader["OrderId"]}: {reader["FullName"]} on {reader["OrderDate"]}");
ถ้า CEK ไม่ตรงกัน จะได้ Msg 33299: Encryption scheme mismatch for columns/variables ... — วิธีแก้คือวางแผนให้คอลัมน์ที่ต้อง join กันใช้ CEK เดียวกันตั้งแต่ตอนออกแบบ schema (บทที่ 3) หรือทำ CEK rotation เพื่อรวมให้ใช้ CEK เดียวกัน

INSERT บนคอลัมน์เข้ารหัส

ทำงานเหมือนคอลัมน์ปกติทุกประการ ตราบใดที่ค่าที่ส่งเข้าคอลัมน์เข้ารหัสมาจาก parameter เท่านั้น

C#
using var command = new SqlCommand(@"
    INSERT INTO dbo.Customer (NationalId, CreditCardNumber, FullName)
    VALUES (@NationalId, @CreditCardNumber, @FullName)", connection);

command.Parameters.Add("@NationalId", System.Data.SqlDbType.Char, 13).Value = "1234567890123";
command.Parameters.Add("@CreditCardNumber", System.Data.SqlDbType.VarChar, 19).Value = "4111111111111111";
command.Parameters.Add("@FullName", System.Data.SqlDbType.NVarChar, 100).Value = "สมชาย ใจดี"; // ไม่เข้ารหัส

command.ExecuteNonQuery();

ผ่าน Stored Procedure (แนะนำสำหรับ Production เพราะ centralize logic และ audit ง่ายกว่า):

T-SQL
CREATE PROCEDURE dbo.usp_InsertCustomer
    @NationalId       CHAR(13),
    @CreditCardNumber VARCHAR(19),
    @FullName         NVARCHAR(100)
AS
BEGIN
    INSERT INTO dbo.Customer (NationalId, CreditCardNumber, FullName)
    VALUES (@NationalId, @CreditCardNumber, @FullName);
END
ทดสอบผ่าน SSMS — literal ใช้ไม่ได้ แม้เปิด "Always Encrypted" ใน connection properties ของ SSMS แล้ว การใส่ literal ตรง ๆ ก็ยัง error เพราะ literal ไม่ถูกส่งเป็น parameter ต้องประกาศเป็นตัวแปรก่อนเสมอ:
T-SQL
-- ผิด: literal ตรง ๆ ไม่ถูกเข้ารหัสให้
INSERT INTO dbo.Customer (NationalId, FullName) VALUES ('1234567890123', N'ทดสอบ');
-- Msg 206: Operand type clash: varchar is incompatible with the encrypted column ...

-- ถูก: ประกาศตัวแปรก่อน — SSMS จะ parameterize ให้อัตโนมัติเมื่อเปิด Always Encrypted
DECLARE @NationalId CHAR(13) = '1234567890123';
INSERT INTO dbo.Customer (NationalId, FullName) VALUES (@NationalId, N'ทดสอบ');

SqlBulkCopy รองรับ Always Encrypted ได้เมื่อเปิด Column Encryption Setting=Enabled ทั้ง source/destination (ห้ามตั้ง AllowEncryptedValueModifications) ส่วน Table-Valued Parameters (TVP) มีข้อจำกัดในหลาย driver เวอร์ชัน — ทดสอบเจาะจงหรือใช้ loop insert ทีละแถวแทนหากพบปัญหา

UPDATE บนคอลัมน์เข้ารหัส

กรณีที่ 1 — filter ด้วยคอลัมน์ที่ไม่เข้ารหัส (แนะนำที่สุด)

C#
using var command = new SqlCommand(@"
    UPDATE dbo.Customer
    SET CreditCardNumber = @NewCreditCardNumber
    WHERE CustomerId = @CustomerId", connection);   // CustomerId เป็น PK ไม่เข้ารหัส

command.Parameters.Add("@NewCreditCardNumber", System.Data.SqlDbType.VarChar, 19).Value = "4222222222222";
command.Parameters.Add("@CustomerId", System.Data.SqlDbType.Int).Value = 42;

command.ExecuteNonQuery();

กรณีที่ 2 — filter ด้วยคอลัมน์เข้ารหัสแบบ Deterministic

รองรับ equality ใน WHERE ได้ตามปกติ driver จะเข้ารหัสค่าพารามิเตอร์ทั้งค่าที่กรองและค่าที่ set ให้อัตโนมัติ:

C#
using var command = new SqlCommand(@"
    UPDATE dbo.Customer SET FullName = @NewFullName
    WHERE NationalId = @NationalId", connection);

command.Parameters.Add("@NewFullName", System.Data.SqlDbType.NVarChar, 100).Value = "สมหญิง ใจดี";
command.Parameters.Add("@NationalId", System.Data.SqlDbType.Char, 13).Value = "1234567890123";

command.ExecuteNonQuery();

กรณีที่ 3 — ห้าม filter ด้วยคอลัมน์ Randomized

WHERE CreditCardNumber = @CreditCardNumber จะ error ทันทีถ้าเป็น Randomized (ไม่รองรับ equality) ต้องหา primary key จากคอลัมน์อื่นก่อน แล้วค่อย update ด้วย PK นั้น:

C#
// ขั้นที่ 1: หา CustomerId จากคอลัมน์ deterministic หรือคอลัมน์ที่ไม่เข้ารหัส
int customerId;
using (var lookup = new SqlCommand(
    "SELECT CustomerId FROM dbo.Customer WHERE NationalId = @NationalId", connection))
{
    lookup.Parameters.Add("@NationalId", System.Data.SqlDbType.Char, 13).Value = "1234567890123";
    customerId = (int)lookup.ExecuteScalar();
}

// ขั้นที่ 2: update ด้วย PK แทน
using var update = new SqlCommand(
    "UPDATE dbo.Customer SET CreditCardNumber = @NewCard WHERE CustomerId = @CustomerId", connection);
update.Parameters.Add("@NewCard", System.Data.SqlDbType.VarChar, 19).Value = "4222222222222";
update.Parameters.Add("@CustomerId", System.Data.SqlDbType.Int).Value = customerId;
update.ExecuteNonQuery();

MERGE / UPSERT

เงื่อนไข ON ที่เทียบคอลัมน์เข้ารหัสต้องเคารพกฎเดียวกับ JOIN ด้านบน (deterministic + CEK/algorithm ตรงกัน) ส่วน UPDATE SET / INSERT ทำงานได้ปกติถ้าค่าที่ใส่มาจาก parameter หรือคอลัมน์ ciphertext ที่เข้ากันได้:

T-SQL
MERGE dbo.Customer AS target
USING (VALUES (@NationalId, @FullName)) AS source (NationalId, FullName)
    ON target.NationalId = source.NationalId   -- deterministic + CEK เดียวกันทั้งคู่
WHEN MATCHED THEN
    UPDATE SET FullName = source.FullName
WHEN NOT MATCHED THEN
    INSERT (NationalId, FullName) VALUES (source.NationalId, source.FullName);
ส่วนที่ 3 — การจัดการคีย์

07การบริหารจัดการคีย์

การจัดการคีย์คือหัวใจของความปลอดภัย หากคีย์หายข้อมูลกู้คืนไม่ได้เลย หากคีย์รั่วไหลข้อมูลทั้งหมดไม่ปลอดภัยอีกต่อไป

หลักการออกแบบ

  • แยกหน้าที่ (Separation of Duties) — ผู้ดูแล key store ควรคนละกลุ่มกับ DBA
  • Least Privilege — จำกัดสิทธิ์ unwrap/decrypt ให้เฉพาะ service account ที่จำเป็นจริง
  • High Availability — key store ต้องมี availability สูงเท่าฐานข้อมูล
  • Disaster Recovery — สำรอง (escrow) คีย์ไว้ในที่ปลอดภัยแยกต่างหาก
  • Audit Trail — บันทึกทุกการ unwrap/decrypt คีย์

สร้าง certificate สำหรับ Always Encrypted บน Windows Server

หากต้องใช้ Windows Certificate Store เป็น key store สำหรับ CMK บน Windows Server (เช่น SQL Server host หรือ application server) ขั้นตอนที่ควรทำมี 4 ขั้นตอนหลัก: สร้าง certificate → ตรวจสอบ thumbprint → import ลงเครื่องอื่น ๆ → ใช้กับ SQL Server

ขั้นตอนที่ 1: สร้าง certificate บน Windows Server

คำสั่งต่อไปนี้จะสร้าง certificate ใหม่ใน Cert:\LocalMachine\My และส่งออกไฟล์ .pfx พร้อม private key:

PowerShell
$certPath = "C:\certs"
New-Item -ItemType Directory -Force -Path $certPath | Out-Null

$cert = New-SelfSignedCertificate `
  -CertStoreLocation "Cert:\LocalMachine\My" `
  -DnsName "sql.contoso.local" `
  -FriendlyName "AlwaysEncrypted-CMK" `
  -KeyExportPolicy Exportable `
  -NotAfter (Get-Date).AddYears(2)

$securePassword = ConvertTo-SecureString -String "P@ssw0rd!123" -Force -AsPlainText
Export-PfxCertificate -Cert $cert -FilePath "$certPath\alwaysencrypted-cmk.pfx" -Password $securePassword
Export-Certificate -Cert $cert -FilePath "$certPath\alwaysencrypted-cmk.cer"

สิ่งที่คำสั่งนี้ทำคือ สร้าง certificate ใหม่ใน Cert:\LocalMachine\My, ส่งออกไฟล์ .pfx ที่มี private key สำหรับใช้กับ CMK และไฟล์ .cer สำหรับแจกจ่ายหรือ import ต่อ

ขั้นตอนที่ 2: ตรวจสอบ certificate และเก็บ thumbprint

หลังสร้างเสร็จ ให้ดึงค่า thumbprint เพื่อใช้ใน SQL Server:

PowerShell
$cert = Get-ChildItem Cert:\LocalMachine\My | Where-Object { $_.FriendlyName -eq "AlwaysEncrypted-CMK" } | Select-Object -First 1
$cert.Thumbprint
$cert.Subject

thumbprint คือค่าที่ SQL Server จะใช้ใน KEY_PATH เมื่อสร้าง Column Master Key (CMK)

ขั้นตอนที่ 3: Import certificate ลงเครื่องอื่น ๆ

หากต้องใช้ certificate นี้ในเครื่องอื่น (เช่น client machine หรือ application server) ให้ import ไฟล์ .pfx ที่ส่งออกมาเข้าสู่ store ดังนี้:

PowerShell
$pwd = Read-Host "กรอก password ของไฟล์ .pfx" -AsSecureString
Import-PfxCertificate -FilePath "C:\certs\alwaysencrypted-cmk.pfx" `
  -CertStoreLocation "Cert:\LocalMachine\My" `
  -Password $pwd

ถ้าต้องการ import ลง Current User แทน Local Machine ให้เปลี่ยนค่าเป็น Cert:\CurrentUser\My

ขั้นตอนที่ 4: ใช้ certificate กับ SQL Server / Always Encrypted

เมื่อสร้าง Column Master Key (CMK) ใน SQL Server ให้ใช้ค่า thumbprint ที่ได้จากขั้นตอนก่อนหน้า:

T-SQL
CREATE COLUMN MASTER KEY MyCMK
WITH (
    KEY_STORE_PROVIDER_NAME = N'MSSQL_CERTIFICATE_STORE',
    KEY_PATH = N'LocalMachine/My/<thumbprint>'
);

ตัวอย่างโครงสร้างคำสั่งสร้าง Column Encryption Key (CEK):

T-SQL
CREATE COLUMN ENCRYPTION KEY MyCEK
WITH VALUES (
    COLUMN_MASTER_KEY = MyCMK,
    ALGORITHM = 'RSA_OAEP',
    ENCRYPTED_VALUE = 0x<hex>
);

ในงานจริง ค่าของ ENCRYPTED_VALUE มักจะถูกสร้างโดย client library หรือ driver ที่รองรับ Always Encrypted (เช่น .NET SqlClient) ไม่ได้เป็นค่าที่คุณสร้างด้วยมือโดยตรง

สำหรับการใช้งานจริง ควรเก็บไฟล์ .pfx ในที่ปลอดภัย หลีกเลี่ยงการเก็บ password ใน script หรือคำสั่งที่เห็นได้ทั่วไป และควรลบไฟล์ชั่วคราวหลัง import แล้วถ้าหน่วยงานไม่ต้องการเก็บไว้ต่อ

Import Certificate และตั้งค่าพื้นฐาน OS บน RHEL9

สำหรับแอปพลิเคชันหรือ client runtime ที่รันบน Red Hat Enterprise Linux 9 และต้องเข้าถึง CMK ที่อ้างอิง certificate-based key store ควรเตรียมระบบให้พร้อมก่อนใช้งานจริง โดยมีขั้นตอนที่แนะนำดังนี้:

  1. ติดตั้งแพ็กเกจพื้นฐาน
    Bash
    sudo dnf update -y
sudo dnf install -y openssl ca-certificates unzip
  2. จัดเก็บไฟล์ certificate ในตำแหน่งที่ปลอดภัย
    Bash
    sudo mkdir -p /opt/alwaysencrypted/certs
sudo cp /path/to/client-certificate.pfx /opt/alwaysencrypted/certs/
sudo chown root:root /opt/alwaysencrypted/certs/client-certificate.pfx
sudo chmod 600 /opt/alwaysencrypted/certs/client-certificate.pfx

    ควรเก็บไฟล์ที่มี private key ไว้ในโฟลเดอร์ที่มีสิทธิ์เข้าถึงจำกัด และควรลบไฟล์ชั่วคราวหลัง import เสร็จแล้ว หากไม่มีความจำเป็นต้องเก็บต่อ

  3. แปลงไฟล์ PKCS#12 เป็น PEM สำหรับ OpenSSL
    Bash
    sudo openssl pkcs12 -in /opt/alwaysencrypted/certs/client-certificate.pfx -clcerts -nokeys \
  -out /etc/pki/tls/certs/client-cert.pem

sudo openssl pkcs12 -in /opt/alwaysencrypted/certs/client-certificate.pfx -nocerts -nodes \
  -out /etc/pki/tls/private/client-key.pem

sudo chown root:root /etc/pki/tls/certs/client-cert.pem /etc/pki/tls/private/client-key.pem
sudo chmod 644 /etc/pki/tls/certs/client-cert.pem
sudo chmod 600 /etc/pki/tls/private/client-key.pem
  4. ติดตั้ง certificate เข้า trust store ของระบบ
    Bash
    sudo cp /etc/pki/tls/certs/client-cert.pem /etc/pki/ca-trust/source/anchors/
sudo update-ca-trust

    หากต้องใช้ certificate chain จาก CA ภายนอก ให้รวมไฟล์ CA bundle ลงในโฟลเดอร์ anchors ก่อนเรียกคำสั่ง update-ca-trust

  5. ตรวจสอบความถูกต้องของ certificate และสิทธิ์ไฟล์
    Bash
    sudo openssl x509 -in /etc/pki/tls/certs/client-cert.pem -noout -subject -issuer
sudo ls -l /etc/pki/tls/private/client-key.pem
  6. ตั้งค่าพื้นฐาน OS สำหรับความปลอดภัย
    • รักษา SELinux ในโหมด Enforcing และตรวจสอบ context ของไฟล์ certificate ด้วย restorecon
    • ใช้ umask 077 หรือ permission ที่จำกัดเมื่อสร้างไฟล์คีย์ใหม่
    • จำกัดสิทธิ์การเข้าถึง private key ให้เฉพาะ root หรือ service account ที่จำเป็นจริง ๆ
    • ถ้ารันแอปพลิเคชันในระบบปิดให้ใช้ service account ที่ไม่ใช่ root เพื่อป้องกันการเปิดเผย private key
    Bash
    sudo restorecon -Rv /etc/pki/tls /opt/alwaysencrypted/certs
  7. สำหรับแอปพลิเคชัน Java/JDBC
    Bash
    sudo dnf install -y java-17-openjdk-devel
sudo keytool -importkeystore \
  -srckeystore /opt/alwaysencrypted/certs/client-certificate.pfx \
  -srcstoretype PKCS12 \
  -destkeystore /etc/ssl/java/cacerts \
  -deststoretype JKS \
  -storepass changeit
    คำแนะนำด้านความปลอดภัย สำหรับ production ควรใช้ password ที่ซับซ้อนและเก็บไว้ใน secret management แทนการ hard-code ลงใน script หรือคำสั่งติดตั้ง

Managed Identity แทนการฝัง secret

C#
var credential = new DefaultAzureCredential();

08การหมุนเวียนคีย์ (Key Rotation)

ระดับคืออะไรความถี่Downtime
CMK Rotationเปลี่ยนกุญแจปกป้อง CEK (ไม่กระทบข้อมูล)ทุก 6–12 เดือนแทบไม่มี
CEK Rotationเปลี่ยนกุญแจเข้ารหัสข้อมูลจริงทุก 1–2 ปี / สงสัยรั่วไหลมี (เว้นแต่ enclave)

ขั้นตอนหมุนเวียน CMK

PowerShell
$database = Get-SqlDatabase -ConnectionString $connStr

# 1) สร้าง CMK ใหม่
$newAkvKey = New-SqlAzureKeyVaultColumnMasterKeySettings `
    -KeyURL "https://mycompany-kv.vault.azure.net/keys/CMK-Prod-2026/<version>"
New-SqlColumnMasterKey -Name "CMK_Prod_2026" -InputObject $database `
    -ColumnMasterKeySettings $newAkvKey

# 2) เข้ารหัส CEK เดิมด้วย CMK ใหม่ (ตอนนี้ถูกเข้ารหัสด้วยทั้งเก่า+ใหม่ = grace period)
Add-SqlColumnEncryptionKeyValue -InputObject $database `
    -ColumnEncryptionKeyName "CEK_Prod" -ColumnMasterKeyName "CMK_Prod_2026"

# 3) หลังยืนยันทุกแอปย้ายไปใช้ CMK ใหม่แล้ว — ลบของเก่า
Remove-SqlColumnEncryptionKeyValue -InputObject $database `
    -ColumnEncryptionKeyName "CEK_Prod" -ColumnMasterKeyName "CMK_Prod"
Remove-SqlColumnMasterKey -Name "CMK_Prod" -InputObject $database
อย่าลบคีย์เก่าทันที เว้นช่วง "grace period" ที่ CEK ถูกเข้ารหัสด้วยทั้งคีย์เก่าและใหม่พร้อมกัน จนกว่าจะมั่นใจว่าทุก client/environment ย้ายไปใช้คีย์ใหม่แล้วจริง (ตรวจสอบผ่าน audit log ของ Key Vault)

หมุนเวียน CEK — In-place ด้วย Secure Enclave

หากเปิดใช้ Secure Enclave ไว้ สามารถทำ in-place rotation ได้โดยข้อมูลถูกถอดรหัส-เข้ารหัสใหม่ ภายใน enclave บนเซิร์ฟเวอร์เอง ไม่ต้องส่งข้อมูลออกไปที่ client เลย เร็วกว่ามาก:

PowerShell
Set-SqlColumnEncryption -ColumnEncryptionSettings $cesc `
    -InputObject $database `
    -RolloverColumnEncryptionKey `
    -TargetEnclaveType "VBS"

Rotation Runbook สำหรับองค์กร

  1. แจ้งล่วงหน้า 2–4 สัปดาห์ก่อนหมุนเวียน CEK เพราะกระทบ downtime
  2. ทดสอบใน Staging ที่ขนาดข้อมูลใกล้เคียง production เพื่อวัดเวลาจริง
  3. Backup ฐานข้อมูลและคีย์เก่าก่อนเริ่ม
  4. CMK ก่อน CEK เสมอ — ให้ระบบใช้ CMK ใหม่มั่นคงก่อน
  5. Monitor error log ของแอปและ SQL Server ระหว่างดำเนินการ
  6. Cleanup หลังพ้น grace period และบันทึกวันที่หมุนเวียนล่าสุด

09Always Encrypted with Secure Enclaves

Always Encrypted แบบดั้งเดิมทำได้แค่ equality comparison บนคอลัมน์ deterministic Secure Enclave (SQL Server 2019+) สร้างพื้นที่หน่วยความจำปลอดภัยบนเซิร์ฟเวอร์เอง ที่แม้แต่ engine/OS/ผู้ดูแลระบบก็มองไม่เห็นข้อมูลภายใน — การคำนวณที่ต้องใช้ plaintext (range comparison, LIKE) จึงเกิดขึ้น ภายใน enclave เท่านั้น

ประเภทเทคโนโลยีใช้กับ
VBSWindows Hypervisor-based isolationSQL 2019+ on Windows, Azure SQL — แนะนำสำหรับติดตั้งใหม่
Intel SGXHardware-based isolation ผ่าน CPUฮาร์ดแวร์ที่รองรับ SGX (ใช้น้อยลงในปัจจุบัน)

ความสามารถเพิ่มเติมที่ Enclave เปิดให้ใช้

  • Rich computations — LIKE, >, <, BETWEEN, ORDER BY บนข้อมูลเข้ารหัส
  • In-place encryption — เข้ารหัสข้อมูลเดิมโดยไม่ต้องดึงออกไปที่ client
  • In-place key rotation — หมุนเวียน CEK ภายใน enclave (บทที่ 8)
  • Encryption type/algorithm change — สลับ randomized ↔ deterministic โดยไม่ดึงข้อมูลออก

เปิดใช้งาน VBS บน SQL Server

T-SQL
sp_configure 'column encryption enclave type', 1;  -- 1 = VBS
RECONFIGURE;
-- จากนั้น restart SQL Server service

ต้องมี Attestation เพื่อยืนยันว่า enclave ไม่ถูกดัดแปลงก่อน client จะส่งคีย์เข้าไป — ใช้ Host Guardian Service (HGS) สำหรับ on-premises หรือ Azure Attestation (จัดการอัตโนมัติโดย Microsoft) สำหรับ Azure SQL

Connection String
Server=myserver;Database=MyProdDb;Column Encryption Setting=Enabled;
Enclave Attestation Url=https://hgs.mycompany.local/Attestation;
Attestation Protocol=HGS;

ตัวอย่าง Query ที่ทำได้เมื่อมี Enclave

T-SQL
-- Range query บนคอลัมน์เข้ารหัส (ทำไม่ได้ถ้าไม่มี enclave)
SELECT * FROM dbo.Employee
WHERE Salary BETWEEN @MinSalary AND @MaxSalary;

-- LIKE บนคอลัมน์เข้ารหัส
SELECT * FROM dbo.Customer WHERE Email LIKE @SearchPattern;
ข้อควรพิจารณา ต้องการ Enterprise/Developer Edition (ไม่รองรับ Standard สำหรับ on-prem), มี performance overhead เพิ่มขึ้น, และต้องวางแผนโครงสร้าง HGS เพิ่มสำหรับ on-premises — Azure SQL จัดการ attestation ให้อัตโนมัติ ใช้งานง่ายกว่ามาก

ข้อจำกัดด้าน Query

การดำเนินการDeterministicRandomized+ Enclave

ข้อจำกัดด้าน Schema/Ecosystem

  • ไม่รองรับ computed column, CHECK/DEFAULT constraint แบบคำนวณบนเซิร์ฟเวอร์, full-text search
  • Foreign key ใช้ได้ถ้าทั้งสองฝั่งเป็น deterministic ด้วย CEK/algorithm เดียวกัน
  • Transactional Replication, PolyBase, Linked Server มีข้อจำกัดมาก — ตรวจสอบ compatibility ก่อนใช้
  • Power BI DirectQuery, SSRS dynamic query builder อาจไม่รองรับ — พิจารณาสร้าง view/extract แยกสำหรับ BI
ไม่ใช่ Silver Bullet Always Encrypted ไม่ป้องกัน SQL Injection บนคอลัมน์ที่ไม่เข้ารหัส และไม่ป้องกันหากแอปพลิเคชันที่มีสิทธิ์ถอดรหัสถูกบุกรุก — ต้องใช้ร่วมกับ network security, authorization, และ secure coding เสมอ (defense-in-depth)

11ผลกระทบด้านประสิทธิภาพ

Overhead หลักเกิดที่ ฝั่ง client/application tier ไม่ใช่ที่ SQL Server engine เพราะการเข้ารหัส/ถอดรหัส ทั้งหมดเกิดที่ driver:

  • CPU ฝั่ง client เพิ่มขึ้น — เข้ารหัส/ถอดรหัสทุก parameter และทุกแถวผลลัพธ์
  • Round-trip เพิ่ม — ครั้งแรกของ query shape ใหม่ต้องเรียก sp_describe_parameter_encryption ก่อน (แต่ถูก cache ไว้หลังจากนั้น)
  • Key store latency — การ unwrap CEK ต้องเรียก Key Vault ทุกครั้งที่ cache หมดอายุ
  • Query plan ด้อยกว่า — ไม่มี useful statistics/range index สำหรับคอลัมน์เข้ารหัส

แนวทางลด Overhead

  1. เข้ารหัสเฉพาะที่จำเป็น — เลือกเฉพาะคอลัมน์อ่อนไหวจริงตามผล data classification
  2. แยกตารางข้อมูลอ่อนไหวสูง — คั่นด้วย 1:1 relationship เพื่อไม่ให้ query ทั่วไปรับ overhead
  3. Deterministic เฉพาะที่ต้องค้นหา — ใช้ Randomized กับคอลัมน์ที่ไม่ query โดยตรง
  4. ทดสอบ Load Test — วัด throughput/latency ก่อน-หลังเข้ารหัสด้วย workload จริง
  5. Region เดียวกัน — วาง Azure Key Vault ใน region เดียวกับแอปพลิเคชันเซิร์ฟเวอร์
  6. Query shape คงที่ — ใช้ parameterize แทนการเปลี่ยนโครงสร้าง query บ่อย ๆ เพื่อให้ metadata cache มีประโยชน์

ขั้นตอนวินิจฉัยปัญหา

  1. ตรวจ Connection String — มี Column Encryption Setting=Enabled หรือเทียบเท่าใน driver นั้นหรือไม่
  2. ตรวจสิทธิ์ Key Store จาก client — certificate มี private key ครบหรือไม่ / service principal มีสิทธิ์ Key Vault หรือไม่
  3. ตรวจ metadata ในฐานข้อมูล — sys.column_master_keys, sys.column_encryption_keys
  4. ทดสอบผ่าน SSMS โดยตรง — เปิด "Always Encrypted" ใน connection properties แล้วรัน query ทดสอบ
  5. เปิด verbose log ฝั่ง driver — ดูรายละเอียดการเรียก key store จริง

Separation of Duties

บทบาทสิทธิ์ที่ควรมีสิทธิ์ที่ไม่ควรมี
DBASchema, performance, backup, metadata CMK/CEKเข้าถึง CMK จริงใน key store
Security/Key Teamดูแล Key Vault/HSM, กำหนดสิทธิ์, rotationsysadmin บนฐานข้อมูล
Application Service Accountunwrap/decrypt เฉพาะที่จำเป็น (runtime)สร้าง/ลบคีย์
Developerเข้าถึง dev/test ที่ใช้คีย์แยกเข้าถึง production key store

Defense-in-Depth — Always Encrypted เป็นแค่หนึ่งชั้น

  1. Network — TLS/SSL, firewall/VNet isolation
  2. At-Rest — ใช้ TDE ร่วมด้วยเสมอเพื่อปกป้อง backup/ไฟล์ฐานข้อมูล
  3. Column — Always Encrypted สำหรับคอลัมน์อ่อนไหวเฉพาะจุด
  4. Access Control — Row-Level Security, Dynamic Data Masking สำหรับคอลัมน์ที่ไม่ต้องเข้ารหัสเต็มรูปแบบ
  5. Application — ยืนยันตัวตนรัดกุม ป้องกัน SQL injection
  6. Monitoring — audit logging, SIEM, anomaly detection บนการเข้าถึงคีย์
Dynamic Data Masking ≠ การเข้ารหัส DDM เพียงซ่อนข้อมูลจาก query ทั่วไป — ผู้มีสิทธิ์สูงยังเห็นข้อมูลจริงได้ ห้ามใช้แทน Always Encrypted สำหรับข้อมูลที่ต้องปกป้องจริงจัง

Checklist ก่อนขึ้น Production

  • ทำ Data Classification และระบุคอลัมน์ที่ต้องเข้ารหัสครบถ้วน
  • เลือก Key Store ที่เหมาะสม (แนะนำ Azure Key Vault/HSM สำหรับ Production)
  • ออกแบบ Deterministic/Randomized ต่อคอลัมน์ พร้อมประเมินความจำเป็นของ Secure Enclave
  • ตั้งสิทธิ์แบบ Least Privilege และ Separation of Duties ครบถ้วน
  • ปรับโค้ดแอปพลิเคชันทั้งหมด (parameterized query, connection string)
  • ทดสอบ Performance/Load Test เทียบก่อน-หลังเข้ารหัส
  • จัดทำแผน Backup/Escrow ของคีย์ และทดสอบกระบวนการกู้คืนแล้ว
  • จัดทำ Runbook สำหรับ Key Rotation และ Incident Response
  • เปิด Audit Logging บน Key Store และตั้งค่า Alert ที่เหมาะสม
  • ตรวจสอบ Reporting/BI/ETL tools ทั้งหมดว่ารองรับ Always Encrypted
